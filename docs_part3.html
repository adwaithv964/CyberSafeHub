5.2 Development Approach

CyberSafeHub followed an Agile-Iterative development methodology, structured into two-week sprints focusing on one major module per sprint. Key development principles:

  • Component-Driven Development: The React frontend was built using a bottom-up approach — atomic components (Button, Input, Card) first, then composite tool components, then full pages.
  • API-First Backend: All backend routes were designed and tested in Postman before frontend integration. This enabled parallel frontend/backend development.
  • Environment Isolation: .env files (never committed to Git) store all secrets. Separate .env.development and .env.production configurations ensure clean environment separation.
  • Git Branching Strategy: Feature branches (e.g., feature/wifi-radar, feature/ai-auditor) are developed independently and merged to main via pull requests after self-review.
  • Progressive Enhancement: Core functionality was implemented first (auth, dashboard, basic tools), then AI features, then educational modules, and finally the admin panel.
  • Error-First Design: All API handlers implement try-catch blocks. Frontend components have error boundary fallbacks. Toast notifications surface errors usefully to the user.
  • Security-First Coding: Backend routes validate Firebase tokens before processing any request. Multer limits file sizes. Rate limiting is applied on public endpoints.

5.3 Database Servers

MongoDB Atlas (Primary Database)

MongoDB Atlas M0 (shared cluster) is used as the primary NoSQL database. The connection is established in server/index.js using Mongoose's connect() method with automatic reconnection.

mongoose.connect(process.env.MONGODB_URI, {
  useNewUrlParser: true,
  useUnifiedTopology: true,
  serverSelectionTimeoutMS: 5000
}).then(() => console.log('MongoDB Atlas connected'))
  .catch(err => console.error('MongoDB connection error:', err));

Security measures: MongoDB Atlas Network Access is configured to allow only the Render server's IP (via IP allowlist). Mongoose sanitizer middleware prevents NoSQL injection. No raw query strings from user input reach MongoDB.

Firebase Firestore (Supplementary)

Firebase Firestore is used for admin role verification and real-time user metadata. The Firebase Admin SDK (server/config/firebaseAdmin.js) is initialised with a service account certificate stored as an environment variable, enabling server-side token verification without client SDK limitations.

In-Memory / Session Storage

Temporary data (scan results, conversion progress, AI conversation history) is held in server memory or passed via Socket.IO events. No persistent session storage is required as Firebase Auth manages session tokens in the browser's localStorage.

// React Context — AuthContext.jsx
const AuthContext = createContext();
export const AuthProvider = ({ children }) => {
  const [user, setUser] = useState(null);
  const [loading, setLoading] = useState(true);
  useEffect(() => {
    const unsub = onAuthStateChanged(auth, (u) => { setUser(u); setLoading(false); });
    return unsub;
  }, []);
  return <AuthContext.Provider value={{ user, loading }}>{children}</AuthContext.Provider>;
};

// Backend route with Firebase token verification — middleware/auth.js
export const protect = async (req, res, next) => {
  const token = req.headers.authorization?.split(' ')[1];
  if (!token) return res.status(401).json({ error: 'Unauthorized' });
  try {
    const decoded = await admin.auth().verifyIdToken(token);
    req.user = decoded;
    next();
  } catch { res.status(401).json({ error: 'Invalid token' }); }
};
